# 筊杯子网格重复渲染修复设计

- 日期：2026-07-11
- 状态：用户已确认方案 A
- 范围：程序化筊杯的 MultiMaterial/SubMesh 分配与渲染验收

## 问题与根因

`VertexData.applyToMesh(mesh)` 会自动创建一个覆盖完整索引缓冲区的默认 `SubMesh`。当前实现随后又手动创建红漆和木质两个 `SubMesh`，最终同一网格存在三层绘制：

1. 默认层用红漆材质绘制全部 1416 个索引；
2. 红漆层再次绘制凸面和侧边；
3. 木质层再次绘制平底。

木质平底因此与默认红漆层共面重复绘制，产生 GPU 深度竞争。黑色放射状三角边与底面三角扇的拓扑一致，是 z-fighting，而不是 Havok 穿模、法线翻转或两片刚体重叠。

## 采用方案

采用方案 A：在创建显式材质分区前释放自动生成的默认 `SubMesh`，随后只创建两个互不重叠的分区：

- material 0：红漆凸面和侧边；
- material 1：木质平底。

两个分区的 `indexStart/indexCount` 必须首尾相接，合计恰好覆盖完整索引缓冲区一次。

## 约束

- 不修改顶点位置、三角形索引顺序、法线、程序化轮廓或材质颜色。
- 不开启双面材质、不关闭深度写入、不使用 polygon offset 掩盖问题。
- 不拆分视觉 Mesh，不修改 `PhysicsAggregate` 的可见凸包来源。
- 不修改质量、摩擦、恢复系数、落地阈值、冲量、摄像机或结果逻辑。
- 不添加依赖。

## 测试与验收

- 回归测试先证明当前网格有 3 个 `SubMesh`，并因覆盖总数大于索引缓冲区长度而失败。
- 修复后要求正好 2 个 `SubMesh`，材质索引为 `[0, 1]`。
- 两个分区连续、无间隙、无重叠，索引覆盖总数等于完整索引长度。
- 现有可见网格与 Havok convex hull 支撑值测试继续通过。
- 运行完整测试、类型检查和生产构建。
- 浏览器重新投掷并人工检查木质平底与斜视角，不得再出现放射状黑边或共面闪烁。

## 完成标准

只有回归测试经历 RED -> GREEN、完整验证通过且最新截图无黑边，才可宣布修复完成。
