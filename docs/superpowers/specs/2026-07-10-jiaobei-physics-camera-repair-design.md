# 筊杯物理时钟、导演镜头与摄像头访问修复设计

- 日期：2026-07-10
- 状态：设计已获用户确认，待实施
- 范围：潮汕筊杯游戏的 Babylon/Havok 掷杯流程、3D 导演镜头和真实摄像头预览

## 1. 问题与证据

当前体验不是单一参数失调，而是三个独立子系统同时存在确定性错误。

### 1.1 物理时间被慢放

`JiaobeiThrower` 使用裸 `requestAnimationFrame()` 调用 `scene.render()`，绕过了 Babylon `Engine.runRenderLoop()` 内部的 `beginFrame()`。因此 `Engine._deltaTime` 保持为 `0`，Scene 将每帧物理步长夹到最小的 `1ms`。

在 60Hz 屏幕上，每 `16.67ms` 真实时间只推进 `1ms` Havok 时间，物理速度约为正常速度的 6%。本应在约 `0.6-1.0s` 内触台的筊杯会被拖到约 `10-16s`。现有流程在 `6s` 时无条件超时，所以可能在杯体尚未真实落定时判定结果。

### 1.2 3D 镜头角度和物理运动失步

当前将 `ArcRotateCamera.beta` 误解为从水平面计算的俯角。Babylon 实际从世界 `+Y` 轴计算 `beta`：当前 `75deg` 与 `69deg` 只相当于约 `15deg` 与 `21deg` 的低机位俯角。

此外，相机只按固定 `800ms` 墙钟动画 target，没有读取两片筊杯的实时坐标。物理慢放时，相机已经移到地面，筊杯仍在高处，导致杯体消失、突然出现、过度近景和两片沿视线重叠。

### 1.3 真实摄像头拒绝后无法恢复

录屏中的右上黑框是浏览器摄像头权限拒绝状态，不是 Babylon 相机。当前 `CameraPosePanel` 只在首次挂载时申请一次权限；拒绝后没有开启或重试入口，也没有区分非安全上下文、iframe policy、用户拒绝、无设备等原因。

识别循环还捕获了首次渲染的 `enabled` 和回调闭包，父组件暂停识别时，循环仍可能继续使用旧状态。

### 1.4 台面和落地判定不可靠

视觉/物理台面只有 `8x8`，而两片带有明确的向外速度和旋转。真实 Havok 随机轨迹可越过 `x/z = +/-4` 后继续下坠，永远无法静止。

当前落地音通过 `mesh.position.y < 0.6` 猜测碰地，但筊杯局部原点在平底处，不同朝向真实接触台面时的原点高度不同。实测首次碰撞可发生在 `y = 0.69-0.72`，因此高度阈值会延迟或漏掉落地事件。

## 2. 目标与非目标

### 2.1 目标

- Havok 时间与浏览器墙钟一致，首次触台约在 `0.6-1.2s` 内发生。
- 两片筊杯始终由独立刚体运动，不重叠出生，不飞出台面后无限下坠。
- 只有两片都真实触台并完成落定后才判定三象。
- 3D 镜头从明确的高机位展示完整抛物线，并实时容纳两片杯体。
- 桌面与窄屏尺寸变化后，drawing buffer、CSS 尺寸和相机投影保持一致。
- 摄像头权限失败后可重试，并向用户显示可操作、准确的原因。
- 掷杯期间姿势识别真正暂停，下一轮可正确恢复。

### 2.2 非目标

- 不把筊杯改成脚本控制的假动画。
- 不通过把重力放大十几倍补偿错误时钟。
- 不增加新的 3D 建模工具或外部运行时依赖。
- 不重做游戏其他页面、文案或三象业务规则。

## 3. 方案选择

采用“真实物理 + 导演镜头 + 可恢复权限”的完整方案。

拒绝仅修改重力/等待阈值的方案，因为它无法修复 Babylon 帧时钟、镜头失步和越界下坠。拒绝脚本动画方案，因为它无法提供用户要求的真实自由落体、碰撞和翻滚。

## 4. 架构与数据流

### 4.1 Babylon 帧循环

`JiaobeiThrower` 保存一个稳定的 render callback：

1. `engine.runRenderLoop(renderFrame)` 负责调度帧并更新真实 delta time。
2. `renderFrame` 依次更新导演镜头、渲染 Scene；Scene 在正确 delta 下推进 Havok。
3. `dispose()` 使用同一个 callback 调用 `engine.stopRenderLoop(renderFrame)`。
4. 重复调用启动方法不得重复注册 render callback。

stage 尺寸由 `ResizeObserver` 观察；尺寸变化后调用 `engine.resize()`，并让导演镜头重新计算安全构图。

### 4.2 物理承接面与碰撞事件

- 保留当前可见神坛台面视觉。
- 将静态物理承接面扩大到至少 `12x12`，优先采用 `14x14` 以覆盖随机滚动余量。
- 保存地面 `PhysicsAggregate`，用于识别具体碰撞对象并在销毁时释放。
- 每片刚体开启 collision callback，只接受首次与地面刚体的碰撞。
- 首次地面碰撞驱动该片的落地音、相机震动和 `landed[index]`，移除 `LAND_Y` 高度猜测。

投掷状态流为：

```text
created -> flying -> piece A/B landed -> settling -> result
                              |              |
                              +-- out-of-bounds recovery
```

正常结果必须满足：两片均收到地面碰撞，并连续一段时间低于稳定速度阈值。安全超时不能把悬空或越界杯体当作正常结果；越界时应恢复到可判定状态或重新执行该次投掷。

### 4.3 导演镜头

相机以两片筊杯的实时包围范围为输入，而不是只使用固定时间曲线。

- `target`：两片世界坐标的中点，Y 在飞行与落定安全范围内平滑限制。
- `radius`：根据两片间距、模型尺寸、相机 FOV、canvas aspect 和右侧浮层安全区计算并平滑变化。
- `beta`：使用 Babylon 正确语义的高机位范围，初始候选约 `0.55-0.75rad`，落定候选约 `0.75-0.95rad`；验收要求相机高度始终高于最高杯体并能看清正反面。
- 掷杯期间 detach 用户控制并清除相机惯性，防止输入和脚本争夺相机。
- 落定后可恢复受限旋转，但禁止 panning 和进入地下。
- 窄屏构图必须将右上摄像头浮层视为不可遮挡安全区；必要时缩小或移出浮层。

### 4.4 真实摄像头访问

`CameraPosePanel` 将摄像头启动从一次性 effect 拆成可重试动作：

1. 首次进入可自动尝试一次，同时始终保留明确的“开启摄像头/重新尝试”入口。
2. 请求前检查 `isSecureContext`、`navigator.mediaDevices` 和可能的权限状态。
3. 对用户拒绝、系统/浏览器策略、无设备和一般初始化失败显示不同提示。
4. 重试前停止并清理旧 stream，成功后重新绑定 video 和识别器。
5. 用 refs 保存最新 `enabled`、`onPrayer` 和不可用回调，避免 RAF 检测循环读取旧闭包。
6. `enabled=false` 时停止 `detectForVideo` 和保持计时；重新启用后从零开始。

## 5. 错误与恢复策略

- Havok/物理初始化失败：保留现有按钮随机降级，不阻塞游戏。
- 单片越界：不得判定杯象；清除本轮刚体并自动安全重掷一次，避免无限地下坠。
- 等待超时但两片已碰地且仅轻微摇晃：允许将极低速度归零后判定。
- 等待超时且任一片未碰地：视为物理异常，进入恢复路径，不返回伪结果。
- 摄像头权限拒绝：保留手动掷杯，同时显示授权说明和重试按钮。
- 非安全上下文或 iframe policy：显示具体限制，不反复弹出无效请求。

## 6. 测试策略

实施严格采用红-绿-重构顺序。

### 6.1 物理与生命周期

- render-loop 契约测试：启动使用 `engine.runRenderLoop()`；重复启动不重复注册；销毁用同一 callback 停止。
- 真实 Havok 时限测试：60Hz 推进下，两片在 `1.2s` 内都产生地面碰撞，且碰撞前不能返回结果。
- 越界回归测试：固定已复现的随机 seed，推进完整流程，断言两片不跌入地面以下并最终进入稳定状态。
- 碰撞事件测试：凸面朝下、原点高于旧 `LAND_Y` 时仍只触发一次落地回调。
- 结算测试：任一片未碰地时，超时不得正常判定；速度重新超过阈值时稳定计数归零。

### 6.2 镜头与响应式

- 相机数学测试：起抛相机高度高于最高杯体，设计俯角达到要求。
- 安全构图测试：桌面和 390px 窄屏下，起抛、最高点、下降和落定状态的两片包围范围都在可见安全区。
- 跟随测试：杯体中点和间距改变时，target/radius 向正确方向平滑更新。
- resize 测试：容器变化后 drawing-buffer aspect 与 CSS aspect/DPR 一致。
- 控制测试：掷杯期间拖动/滚轮不得改变导演镜头。

### 6.3 摄像头

- 第一次 `NotAllowedError` 后显示重试；随后授权并重试可恢复预览。
- `enabled` 从 true 变为 false 后，`detectForVideo` 调用停止；恢复 true 后才重新开始。
- 非安全上下文、无 mediaDevices、无设备和用户拒绝映射到不同提示。
- 卸载或重试时所有旧 tracks 均被停止。

## 7. 验收标准

- 浏览器 60Hz 下物理不再慢放；首次碰地在 `1.2s` 内。
- 正常投掷不会依赖 `6s` 强制超时，且结果不会在两片真实碰地前出现。
- 固定回归 seed 不越界、不落入地下；多 seed 验证无越界结果。
- 录屏中能连续看见起抛、最高点、下降、碰撞和落定，两片不沿视线重叠成一个轮廓。
- 桌面与 390px 视口不裁掉任一片筊杯，画面不因旧 drawing buffer 被压扁。
- 摄像头拒绝后有明确重试路径；授权后无需离开游戏即可恢复预览。
- `npm test`、`npx tsc --noEmit` 和 `npm run build` 全部通过。
